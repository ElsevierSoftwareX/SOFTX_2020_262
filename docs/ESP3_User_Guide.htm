1.    Introduction

a.      What is ESP3?

ESP3 (Echo Sounder Package 3) is an open-source software package with Graphical User Interface for visualising and processing fisheries acoustics data from single-beam and split-beam echosounders. It was initiated and is still mainly developed by the Fisheries Monitoring & Acoustics group, Fisheries National Science Centre, NIWA (Wellington, New Zealand).

ESP3 is built in priority for SIMRAD EK60 and EK80 data (.raw format) but also supports a few other formats to some extent: NIWA CREST, Furuno FCV-30, and ASL AZFP. The focus of the software is the processing of fisheries acoustic surveys, with attention to reproducibility and consistency. ESP3 offers a range of standard and novel data processing operations including calibration, echo-integration, automated bad pings identification, automated bottom detection, single targets identification and tracking, school detection, multi-frequency analysis, etc.

b.      ESP3 version and documentation

This documentation is valid for ESP3 version 1.8.4 (released June 2020).

This documentation may be packaged with a later version of the software. The documentation update tends to lag behind ESP3 because the software is under very active development and releases of compiled versions are very frequent to make the latest algorithms available to users. If you are using a later version of ESP3, then:

·         Screenshots in this documentation may not look the same as in the software.

·         Some algorithms available in the software may not be described in the current documentation.

·         Some software behaviour described in this documentation may not apply.

ESP3 is written in MATLAB under Git version control. With the source code, it can be run from a standard MATLAB environment provided the appropriate version and toolboxes are installed.

The ESP3 source code is hosted at https://bitbucket.org/yladroit/esp3.

Using the source code version of ESP3 requires MATLAB R219b and the following MATLAB toolboxes:

·         Curve Fitting Toolbox

4.    Echogram display options

a.      Echogram Display

The primary visual component of ESP3 is the Echogram Display shown as a tab in the Main Panel (Figure 13). An echogram is a colour-coded image of the recorded acoustic data where each column represents one successive ping with the beginning of the recording at the top.

Figure 13: Echogram Display in the Main Panel, and Overview Display in the Display Option tab of the Control Panel

b.      Frequency channel and data type

An echogram only shows one data type for one frequency channel at a time. Other data types and other frequency channels exist and can be displayed. The name of the Echogram Display tab in the Main Panel indicates which data type and frequency channel are being displayed (Figure 13).

If it was chosen at the file loading stage, the frequency channel displayed by default is 38 kHz. To select one of the other frequency channels that had been loaded:

·         In the Control Panel, Display Option tab, use the drop-down menu next to Chan. (Figure 14).

·         Alternatively, use the shortcut F to toggle between frequency channels.

The data type displayed by default is the volume backscattering strength  in dB re 1 m-1 (MacLennan et al. 2002). It is calculated from the raw data level recorded in the files using the calibration parameters and absorption, which can be respectively modified in the Calibration and Environment tabs of the Control Panel. The other data types available directly after loading the file are, in the case of EK60 data:

·         Power: the raw level of the data samples in the files (in dB), from which  is calculated.

·         Angle Across and Angle Along: angles in the across- and along-track directions (in degrees) for each data sample.

·         TS (uncomp.): Target Strength in dB re 1 m2 (MacLennan et al. 2002), also calculated from the Power, calibration parameters, and absorption. Note that this value is not compensated for antenna directivity (i.e. beam pattern), hence the uncomp. in the variable name.

To select a different data type:

·         In the Control Panel, Display Option tab, use the drop-down menu next to Data (Figure 14).

·         Alternatively, use the shortcut E to toggle between data types.

Note that some algorithms create new data types, which are then accessible in this drop-down menu.

d.      X and Y axes grids and units

The native unit for the echograms Y-axis is the time since the beginning of the record (with 0 at the top and increasing downwards). However, those values are converted by default to range in metres, using the sound speed parameter displayed in the Environment tab of the Control Panel.

Important note: On the Echogram Display, the Y-axis represents the range from the transducer face, not the depth (range from the surface).

The orientation of the Y-axis can be reversed to aid the interpretation and analysis of data collected using moored echosounder. To change the orientation of the Y-axis, go to the Menu Bar, expand the Display menu, and check Reverse Y-axis.

The native unit for the echograms X-axis is the number of pings since the start of the record (with 0 on the left and increasing rightward). Since ping time and vessel navigation data are available for each ping, these ping numbers can be converted in units of time and distance, respectively seconds and metres since the start of the record. Since vessel speed and/or ping rate can change through a recording, the graph grid in the X-axis can be irregular if using those units.

The default unit for the X-axis is metres. To change this to either seconds or pings, use the drop-down menu next to X grid in the Display Option tab of the Control Panel (Figure 15).

The grid steps for both axes can be modified by entering a new value in the box next to the X grid or Y grid label, respectively (Figure 15).

6.    Annotating and Cleaning

A main purpose of ESP3 is echo integration, that is, the calculation of a total area or volume backscattering coefficient for a set of samples over a range of pings. This requires the dual capability of defining the areas of the echogram to be considered for integration (for example, a fish school) and defining the samples within these areas that one wish to exclude from the calculation (for example, the bottom echo, or noise). The first process is annotating, and the second process is cleaning.

In ESP3, annotating is realized by defining Regions (of interest). There are several approaches to defining Regions, depending on the user objectives.

In ESP3, cleaning is realized depending on the three types of samples that one may wish to exclude from the calculation:

1. The samples making up the bottom echo. This is because they are so energetic that even one sample can dramatically skew the results. In ESP3, one can define a sample in every ping below which all data samples are to be ignored in the integration. This lower bound for valid data is termed the Bottom Line. ESP3 includes algorithms to automatically or semi-automatically set this Bottom Line as well as a range of tools to examine the results of this algorithm and to manually set, modify, shift, remove and save the Bottom Line.

2. The corrupted samples, showing a power level that is either too low (i.e. signal dropout) or too high (i.e. noise burst) compared to the expected signal. There are many causes to these corrupted samples, such as bubbles at the transducer face, mechanical vibrations, or interferences. In ESP3, they are called Bad Pings when they affect entire pings, and Spikes when they only affect a few consecutive samples in a given ping. ESP3 includes automated and semi-automated algorithms to identify Bad Pings, or consecutive samples as spikes, so that they are excluded from echo-integration.

3. The samples composing a target in the water-column that is not of the type we wish to include. In ESP3, one would define a Region around these samples and label it as Bad Data for exclusion from echo-integration.

Setting the Bottom Line, flagging Bad Pings and Spikes, and defining Regions for integration thus form the core of data pre-processing in ESP3, usually in that order. Figure 25 shows a typical aspect of ESP3 with those elements defined.

Note that the Bottom Line, the Bad Pings and Spikes flags, and the Regions are defined separately for each frequency channel in a file (Figure 26), although there are options to copy those across to the other channels.

c.      Region parameters

Every Region has the following parameters, composing the columns in the Region List tab of the Control Panel (Figure 40). At the exception of the Unique ID parameter, they can all be modified in this table.

·      Name: There are three methods to create Regions, and the Name parameter originally refers to which method was used to create the Region: User defined for Regions created manually by interacting with the Echogram Display using any of the Region creation tools available in the Create Region mode from the Tools Bar (shortcut 4); WC for Regions created using the Create WC Region tool (in Menu > Tools > Region Tools); or School for Regions created using the School Detection algorithm (in Algorithms Panel > School Detect./Classif tab). Refer to the next sections for details about each of these methods of creation. The Name parameter can be modified after the Region has been created.

·      ID: Each Region is given an incrementing number as an ID at creation. This ID parameter is displayed in parentheses on the corresponding Region in the Echogram.

·      Tag: A Region tag is a string of characters used to label all Regions that need to be integrated together. Species names or acronyms are typically used for tags. The Tag parameter is displayed as a label on the corresponding Region in the Echogram.

·      Type: Data or Bad Data. While Regions were designed to identify the areas of interest for echo integration, it is also possible to use them to flag areas to exclude from the echo integration. A Region with type Bad Data will be excluded from echo integration calculations just like samples under the Bottom Line, samples from pings flagged as Bad Pings, and samples flagged as Spikes. Any normal Region will have Data as Type.

·      Cell Width, Width Unit, Cell Height, Height Unit: While one may wish to perform the echo integration on all samples within the entire Region  that is, to return a single backscattering coefficient for the Region  ESP3 allows a finer level of detail by binning samples within cells of set width (in number of pings, seconds, or meters) and set height (in meters). The Region cell is thus the actual unit area of interest during integration, and echo integration of a Region is simply the sum of the results of the echo integration on its constituent cells.

·      Reference: Surface, Bottom, or Transducer. Given the binning of a Region in cells, as discussed above, it is necessary to define the reference point for the bins. The horizontal reference will always be the first ping in the Region, which implies that the last bin of the Region may be less wide than all others. The Reference parameter is the vertical reference, which can be the Surface, Bottom Line, or Transducer face.

·      Unique ID: A Unique ID is given to a Region after creation and is used by ESP3 to distinguish Regions from one another. This parameter cannot be changed.

e.      Water-Column (WC) Regions

WC Regions are Regions spanning the entire set of pings in the Echogram, only bounded to an interval of either depth (from the surface), range from the transducer face, or height above the Bottom Line (seabed). They are generated by going to the Menu Bar, selecting the Tools menu, then Regions Tools, then Create WC Region. A window then appears where parameters can be set (Figure 42).

Since they span the entire set of pings, WC Regions can be thought as analog to user-defined Region of Vertical shape, except with vertical bounds informed numerically rather than manually. However, the comparison is only valid for WC Regions created with Transducer as a Reference (Figure 42), since the Echogram Display is referenced on its Y-axis to the transducer face.

When a WC Region is defined with the Bottom as a reference line (Figure 43), the Region thus created will be shaped following the Bottom Line (Figure 45). These Regions will not appear rectangular on the Echogram Display.

Where user-defined Regions are created to target schools and other targets of interest on the Echogram Display, WC Regions are created to run echo integration on the entire file, at set depths of interest, or set heights from the bottom. Thus, WC Regions are most commonly defined with either Surface or Bottom as their Reference.

The Compute on Denoised data parameter (denoised) controls whether the algorithm is applied to the normal TS (uncompensated) data (denoised=0) or after it was denoised, aka Denoised Sp (denoised=1).

Whatever the choice, data are transformed into surface-equivalent backscatter for the rest of the algorithm ( where TS is the uncompensated Target Strength and R is the range).

·         Step 2: Data cropping

Data are cropped in depth, as controlled by the Minimum Depth parameter (r_min, 0 by default) and the Maximum Depth parameter (r_max, infinite by default). Ringdown is also removed from the data.

Pings for which the maximum BS level is less than the BS Threshold parameter (thr_bottom, -35 by default) are discarded from the algorithm. This threshold is to prevent detecting a bottom in pings where the bottom was out of data range, or data where cavitation or other technical issue prevented the recording of a strong bottom echo.

Changing this parameter will therefore not change the location of the detected bottom in any given ping, but which pings it is applied to.

Use a sufficiently low BS Threshold to force the algorithm to be applied to all pings even if they show a weak bottom echo.

Make BS Threshold higher if you wish to prevent the algorithm to be applied to pings that dont have a clear bottom echo.

·         Step 3: Finding a first bottom sample

To find a bottom sample, the algorithm first limits the data analysed to the samples that have a BS value that exceeds a threshold defined by the maximum BS in a ping plus the Echo Threshold parameter (thr_echo, -35 by default). For example, if the maximum BS value in a ping is -28.7 dB and the parameter is set at its default value of -35 dB, the algorithm will proceed with samples with a value of at least -28.7-35 = -63.7 dB.

Changing this parameter does change the location of the bottom sample in any given ping, but its influence is limited unless there are strong echoes in the water column above the bottom.

Increase the value from the default -35 dB to force the algorithm to omit strong water-column echoes.

·         Step 4: Recursive back-stepping

The algorithm then searches within one pulse length above the detected bottom sample if there are any sample for which the difference in BS with the bottom sample is less than the Back-step Threshold parameter (thr_backstep, -1 by default). If this occurs, this sample becomes the new detected bottom sample and the process is repeated. The recursive process stops when the difference in BS between the bottom sample and the sample above at one pulse length is greater than the Back-step Threshold parameter.

This process allows finding the closest sample from the initial detection that represents a significant step-up in signal strength.

This is the most sensitive parameter of the list. Modify carefully.

Use 0 to prevent the algorithm from back-stepping.

Lower this value to -3, or -5, or further, when the signal above the bottom is rather strong (sidelobe artefact, for example) and you want your bottom sample to be defined above this fuzzy zone.

·         Step 5: Bottom shifting

The algorithm steps back the bottom even further by the constant distance informed by the Shift Bottom parameter (shift_bot, 0 by default).

 

·         Step 6: Interpolation

If the interp_method parameter was informed, the algorithm finishes with interpolating the values where none could be set earlier, using the methodology chosen.

·         Sub-algorithm #1: The undefined bottom-line analysis

This sub-algorithm flags as bad pings all pings for which the bottom-line is undefined, unless they come in a block of six or more consecutive pings (Figure 49). This algorithm has no controllable parameter.

Figure 49: Result of the application of the undefined bottom-line analysis sub-algorithm of the Bad Pings Detection algorithm, targeting pings with undefined Bottom Line.

·         Sub-algorithm #2: The ringdown analysis

Under normal conditions, the ringdown (i.e. the recording of the pulse as it is being transmitted) is very stable. Abnormal values are indicative of issues in transmitted power, often due to cavitation. This sub-algorithm analyses the ringdown for abnormal power level.

The Ringdown Analysis Flag parameter (Ringdown_std_bool, 1 by default) controls whether the sub-algorithm is applied (Ringdown_std_bool=1) or not (Ringdown_std_bool=0).

The sub-algorithm finds and extracts in each ping the raw level sample (Power) corresponding to approximately a third of the pulse length. Two analyses are then performed on this sample over all pings. First, the algorithm computes a histogram of all samples, takes the maximum value as the ringdown nominal value, and flags all samples that differ from the nominal value by more than twice the Ringdown Deviation Threshold parameter (Ringdown_std, 0.05 by default). Then, to account for the possibility of the transmitted power changing over time, a similar analysis is applied in which a sliding subset of samples are considered instead of the full set. Histograms are computed on those subsets, nominal values are defined as the average of the highest values in each histogram, and the algorithm flags all samples than differ from the nominal value by the Ringdown Deviation Threshold parameter.

 

Accordingly, the lower the Ringdown Deviation Threshold parameter, the more sensitive is this algorithm, and the more pings will be flagged as bad pings.

 

Figure 50: Example of results from the ringdown analysis

·         Sub-algorithm #3: The bottom-echo backscatter analysis

A common symptom of a ping that was not transmitted correctly throughout the water column is a bottom echo that is significantly weaker or stronger than normal. This sub-algorithm analyses the bottom echo for such abnormal values.

The Bottom Echo Analysis Flag parameter (BS_std_bool, 1 by default) controls whether the sub-algorithm is applied (BS_std_bool=1) or not (BS_std_bool=0).

The algorithm starts by transforming data into pseudo-backscatter surface-equivalent backscatter ( where  is the volume backscattering strength and  is the range). It then calculates the average value (in linear space) of the bottom echo approximated as all samples between the bottom detect and the sample at the range corresponding to 110 % of the bottom detect range. The algorithm then computes a running average over two windows of 3 and 5 pings, and flag as bad pings those whose moving average exceeds the Bottom Echo Deviation Threshold parameter (BS_std, 6 by default).

Accordingly, the lower the Bottom Echo Deviation Threshold parameter, the more sensitive is this algorithm, and the more pings will be flagged as bad pings.

 

Figure 51: Example of results from the bottom echo backscatter analysis

·         Sub-algorithm #4: The above-bottom-echo backscatter analysis

This algorithm analyses the average backscatter data in the water column between the end of the ringdown and the bottom detect for abnormally strong or abnormally weak level.

The Above-Bottom-Echo Analysis Flag parameter (Above, 1 by default) controls whether the sub-algorithm is applied (Above=1) or not (Above=0).

The algorithm starts from the same surface-equivalent backscatter calculated previously. It then calculates the average value (in dB space) of the section of data between the ranges corresponding to the end of the ringdown and the bottom detect, if the algorithm is applied to the entire data. If applied to a Region, the entire section of data above the bottom detect will be considered. A peak detection algorithm is then applied to identify individual pings (or groups of 2 to 3 pings) with a level higher than its neighbours by more than the Above-Bottom-Echo Deviation Threshold parameter (thr_spikes_Above, 3 by default). The process is repeated to identify negative peaks.

 

Accordingly, the lower the Above-Bottom-Echo Deviation Threshold parameter, the more sensitive is this algorithm, and the more pings will be flagged as bad pings.

 

·         Sub-algorithm #5: The below-bottom-echo backscatter analysis

 

This algorithm analyses the average backscatter data in the water column between the end of the bottom echo and the first multiple return for abnormally weak level.

The Below-Bottom-Echo Analysis Flag parameter (Below, 0 by default) controls whether the sub-algorithm is applied (Below=1) or not (Below=0).

The algorithm starts from the same surface-equivalent backscatter calculated previously. It then calculates the average value (in dB space) of the section of data between the ranges corresponding to 110% of the bottom detect range and the first bottom multiple, if the algorithm is applied to the entire data. If applied to a Region, the entire section of data below the range corresponding to 110% of the bottom detect range is considered. The peak detection algorithm is then applied to identify individual pings (or groups of 2 to 3 pings) with a level lower than its neighbours by more than the Below-Bottom-Echo Deviation Threshold parameter (thr_spikes_Below, 3 by default).

 

Accordingly, the lower the Below-Bottom-Echo Deviation Threshold parameter, the more sensitive is this algorithm, and the more pings will be flagged as bad pings.

 

Note that this sub-algorithm is very sensitive if there are few data below the bottom echo.

11.           Handling multiple files

a.      Survey and layers

A typical acoustic survey results in the recording of several raw data files, acquired for various purposes, areas, stations, repeats, etc., which implies a need to identify files and differentiate between data records. In addition, it is common that data recorded for a specific purpose do not exactly span a single, entire file. For example, a long individual acoustic transect may span several consecutive files, or a single file may contain several short transects requiring separate processing.

To manage this, ESP3 provides a system of metadata that makes two assumptions:

·         All files within a same folder were acquired as part of the same survey and are meant to be processed together. Data in different folders are considered separate surveys.

·         Individual files are not the unit source of data, but instead any consecutive files or sections of files acquired for the same purpose, as identified by their common metadata. These are termed layers.

b.      Metadata attributes

A survey is identified by two metadata attributes, which thus apply to all layers within the survey:

·         Survey name: A free-text attribute normally used for the full name of survey, including for example year, location, type of survey or target species.

·         Voyage: A free-text attribute normally used for the voyage code.

A layer in a given survey is identified by four metadata attributes:

·         Snapshot: A numerical attribute (integer) normally used to identify the survey repeat to which the layer belongs.

·         Type: A text attribute following a controlled vocabulary, normally used to identify the activity or purpose behind the acquisition of a layer. The controlled vocabulary is defined in the file types.xml located in the config folder of the ESP3 application (normally, C:\Program Files\ESP3_ver_X.X.X\config). It is thus editable by modifying the XML file. The default vocabulary is:

o   Acoustic: General purpose type, used for acquisition of an acoustic transect.

o   Whales: Whales Observations.

o   RMT: RMT Trawl.

o   Trawl: Acoustic file recorded during trawling.

o   Steam: Acoustic file recorded while steaming between stations.

o   Mooring: Mooring deployment.

o   Calibration: Calibration layer.

o   Deployment: Instrument deployment.

·         Stratum: A free-text attribute normally used to identify the geographical stratum where the layer was acquired.

13.           Survey processing with the scripting module

a.      Introduction to scripts

The primary purpose of ESP3 is to produce combined results of the echo-integration of many acoustic data files for the estimation of fish biomass in a survey. This is done through the Scripting Module, which allows the user to run batch processing on several transects, with set algorithms and set parameters of his or her choice, all described on a text file following a predefined format, coded in XML, and called script (Figure 69).

In summary, a script typically contains:

·         The ESP3 general parameters (such as sound speed, absorption coefficient, or echo-integration parameters);

·         The algorithms to be applied and their parameters;

·         The version of the Bottom Line or Bad Pings flags to use;

·         The calibration values;

·         The transects to be processed (identified by their metadata attributes); and

·         The Regions within which data is to be echo-integrated.

In survey processing, ESP3 echo-integrates data at the fundamental level of a vertical slice of transect, defined by the vertical_slice_size and vertical_slice_units parameters.

When a script is run, ESP3 uses the information in the file to:

·         Load the desired data (transects, Bottom Line, bad flag transmits, Regions, etc.);

·         Set the desired parameters;

·         Run the desired algorithms in the order they are listed;

·         Performs an echo-integration of all data included in the desired Regions at the exclusion of bad data (under the Bottom Line, flagged as bad transmit, bad data Region, or out of desired depth bounds), for each vertical slice of each transect;

·         Combine the results per vertical slice into results per transect, and then into results per stratum; and

·         Export these results in a spreadsheet file (.xlsx).

Results will be generated at the three levels mentioned: per transect slice, per transect, and per stratum.

d.      Script format

                                                   i.            Root organization

An ESP3 script is organized as follows (Figure 73):

·         The file starts with the XML prolog <?xml version="1.0" encoding="UTF-8"?>

·         The root element is called <survey_processing>

·         This root element has six child elements:

o   <survey ... /> is a self-closing tag with attributes providing information (metadata) about this script and the data it processes.

o   <options ... /> is a self-closing tag with attributes defining ESP3 parameters.

o   <cal ... /> is a self-closing tag with attributes defining the calibration parameters for one frequency. There can be several of these tags  one for each frequency.

o   <regions_WC ... /> is a self-closing tag with attributes defining one WC Region. This element is optional.

o   <algos> is a regular element whose contents are elements for each algorithm to be applied.

o   <snapshot ...> is a regular element whose contents will list the transects to be processed.

Figure 73: The general organization of an ESP3 script

                                                 ii.            The <survey ... /> element

The <survey ... /> element is a self-closing tag with attributes providing information (metadata) about this script and the data it processes.

The attributes available are (Figure 74):

·         Areas: Geographical area where survey took place.

·         Author: Script author.

·         Comments: Free-text field for comments, typically summarizing the batch processing being applied.

·         Created: Date of creation for script, in the dd-mmm-yyyy HH:MM:SS format.

·         Main_species: tag(s) for the species being targeted.

·         Script: file where the script is saved

·         SurveyName: Full name of the survey.

·         Title: title given to the script (describing the processing)

·         Voyage: name of the associated voyage/trip/mission

·         XmlId: not used at the moment.

Although none of this information affects the results of the batch processing, it is strongly recommended to be completed carefully to avoid confusion between the many scripts that are typically generated for any survey (see Figure 71 for example).

If an attribute is not specified, ESP3 will use empty text as the default value for it.

 

Figure 74: An example of complete <survey ... /> element in an ESP3 script

                                               iii.            The <options ... /> element

The <options ... /> element is a self-closing tag with attributes defining the ESP3 parameters for batch processing.

The attributes of this element and default values are (Figure 75):

·         Absorption: Absorption to be used for the survey (db/km). Default: NaN: will use file absorption. If multiple frequencies are loaded, specify absorption for each frequencies separated by ;.

·         BadTransThr: Threshold on Bad Pings percentage. Transect with more than BadTransThr% of Bad Pings will be removed from further analysis. Default: 100 (keeps all transects).

·         CopyBottomFromFrequency: Specified as one of the loaded frequencies. If not left to 0, it will copy from the bottom from the specified frequency to all other ones. Default: 0.

·         Denoised: Use denoised  for integration if set to 1. Default: 0.

·         DepthMax: Maximum depth for integration in meters. Default: inf.

·         DepthMin: Minimum depth for integration in meters. Default: 0.

·         Es60_correction: if specified, will use this value (in dB) as reference value to correct for triangle wave errors for transect that are too short for it to be determined automatically (only useful for ES60 and ES70 data). Default: NaN.

·         ExportRegions: if set to 1, will export each integrated Regions as a separate file with cell per cell integration in addition to general results. Default: 0.

·         ExportSlicedTransects: if set to 1, will export each sliced transect as a separate file with cell per cell integration in addition to general results. Default: 0.

·         FrequenciesToLoad: list of frequencies to load including the main frequency. Put in ascendent order (Hz), separated by ;, i.e. 18000;38000;70000. Default: .

·         Frequency: Main frequency for the analysis (Hz). Default: 38000.

·         Horizontal_slice_size: Horizontal (according to depth) slicing size for transects (in meters). Default: 10.

·         Vertical_slice_size: Horizontal slicing size for transect. Default: 100.

·         Vertical_slice_units: Units for horizontal slicing. Default: pings. Can be seconds, meters,or pings.

·         Motion_correction: If set to 1, will apply a motion correction algorithm before integration (Dunford, 2005). Default: 0.

·         Remove_ST: If set to 1, will remove detected single targets from integration. Default: 0.

·         Remove_tracks: If set to 1, will remove tracked targets from integration. Default: 0.

·         SaveBot: If set to 1, will save bottom as .xml files. Default: 0.

·         SaveReg: If set to 1, will save Region as .xml files. Default: 0.

·         Shadow_zone: if set to 1, will apply a Shadow/Dead zone correction based on height above the seafloor defined in Shadow_zone_height parameter. Computed using the method of R. Barr (NIWA, Wellington, unpublished report). Default: 0.

·         Shadow_zone_height: Height above the seafloor (in meters) used to estimate the  value to be extended in the estimated shadow/dead zone. Default: 0.

·         Soundspeed: Soundspeed to be used in survey analysis (m/s). Default: nan.

·         ClassifySchool: If set to 1, will apply a classification to Regions and change their tag accordingly. The classification is a tree-based classification following the classification tree described in ClassificationFile . Default: 0.

·         ClassificationFile: Classification file to be used. Default: D:\Docs\MATLAB\ESP3\config\classification\classification.xml.

·         SvThr: Threshold on  values. Values below this will be set to empty water (-999 dB). Default: -999.

·         Use_exclude_regions: If set to 0, Regions defined as Bad Data will be included in the analysis. Default: 1.

If an attribute is not specified, ESP3 will use the default value for it.

Note that any of these parameters can be redefined in the <snapshot ...> element, allowing for different snapshots to be processed with different parameters before the results are combined.

                                               vi.            The <algos> element

The <algos> element is a regular element whose contents are elements for each algorithm to be applied (Figure 78). These child elements are self-closing tags flagging for the application of the corresponding algorithm:

·         <BottomDetection ... /> for the Bottom Detection V1 algorithm.

·         <BottomDetectionV2 ... /> for the Bottom Detection V2 algorithm.

·         Algorithms will be applied in the order in which they are listed under the <algos> element.

·         It is possible to apply the same algorithm as many times as desired by simply repeating the element.

·         Scripts allow the application of algorithm to define Bottom Line, Bad Pings and Regions and save the results (in the options element, attributes SaveBot and SaveReg).

·         If you run a script calling for the application of these algorithms and the saving of their results, your manually edited Bottom Line, Bad Pings and/or Regions will be ignored and overwritten.

                                             vii.            The <snapshot ... > element

The <snapshot ...> element is a regular element whose contents lists the transects to be processed. It is organized with nested elements <stratum ...> and <transect ...> (in that order) with the attributes of each element identifying the snapshot number, stratum and transect number of the transect to be processed (Figure 79). To process several transects within a stratum, several strata within a snapshot, or several snapshots, simply add these elements while following the nesting hierarchy (Figure 79).

The attributes of <snapshot ...>, <stratum ...> and <transect ...> allow identifying the transects to be processed through their metadata (see section on metadata attributes):

·         number in <snapshot ...> is the snapshot number.

·         folder in <snapshot ...> is the folder where the data for this snapshot are.

·         name in <stratum ...> is the stratum name.

·         type in <stratum ...> restrict the type of transects to this one.

·         design in <stratum ...> is the stratum design, either hill or flat (default flat).

·         number in <transect ...> is the transect number.

Finally, the <transect ...> element has two necessary children elements <region ...> and <bottom ...>. Both those children have a ver attribute that define the version of the bottom or Region that will be loaded. If version is set to -1, the bottom/Region will load the .xml file, if set to 0, it will load the latest version available, either from the bottom/Region database or from the .xml file if the database is empty. If set to N, it will load the database version N for those. Note that if you are loading a bottom and then apply a bottom detection algorithm, the bottom that will be applied is the one from the bottom detection. For Regions, if you are loading saved schools and then apply a school detection algorithm, the schools that will be kept will be the newly detected schools.

 <region ...> defines the Region(s) to be echo-integrated within the transect. Use attribute name, ID, or tag to identify the Region(s) of interest. Several names, IDs or tags can be combined by separating the identifier with a semi-column (for example: tagEUP;MMU will include all Regions tagged EUP or MMU).

15.           Calibration

CW mode:

Open loading a file, ESP3 reads the raw Power data and calculates the backscattering coefficient  based on the two calibration parameters Gain (dB) and Sa Corr (dB) displayed in the Calibration tab of the Control panel (Figure 80). ESP3 recalculates  automatically if these values are modified.

These parameters must be correct to produce a quantitatively correct  and thus quantitatively correct echo-integration results. Thus, these values should always be verified before running an echo-integration calculation. If using a script, they can be specified then and do not need to be specified as such.

Figure 80: The "Calibration" tab in the Control Panel. The calibration parameters Gain (dB) and Sa Corr (dB) are automatically used to calculate the  and   displayed. All other settings and buttons can be used to determine new calibration parameters, and discussed in a later section

In order to recalculate those two parameters using calibration data, use the following sequence.

First, set appropriate parameters in the Environment sub-panel. Usually:

·         Set the temperature in Celsius degrees (at NIWA, take mean temperature from table 2 in calibration analysis document) and the salinity in PSU (at NIWA, take the mean salinity from table 2 in calibration analysis document).

·         ESP3 calculates attenuation and sound speed using the method indicated in the drop-down menu (Doonan et al 2003, or Francois and Garrison 1982). Use the values calculated or overwrite them by checking the check box next to those values. At NIWA, table 2 in calibration analysis document provides sound absorption and sound speed values.

·         Click Apply env.  to record these parameters and recompute the  values.

Figure 81: The "Environnement" tab in the Control Panel.

Next, define one or several Regions around the sphere echo in the echogram.

Next, run the calibration:

·         Click on Process TS Cal  in the Calibration tab.

·         The results are in the command window and in a text file exported automatically in the same folder as raw data, and titled cal_log_XXkHz_YYYMMDD_hhmmss where XX is the frequency and YYYMMDD_hhmmss is the date and time at which the calibration was run (e.g. cal_log_18kHz_20200323_084444 for 38 kHz tun on the 23/03/2020 at 08:44:44).

·         The two major results are:

o   The gain, found at line 16 in the text file: So the calibrated G_o = XXX dB (mean method).

o   The Sa Corr, found at line 26 in the text file: So sa correction = XXX dB.

·         Other values relevant to NIWA procedure:

o   Beamwidth (°) along: Fore/aft beamwidth = XXX degrees (line #28).

o   Beamwidth (°) athwartship: Port/stbd beamwidth = XXX degrees (line #30).

o   Beam offset (°) along: Fore/aft offset = XXX degrees (to be subtracted from EK60 angles) (line #29).

o   Beamwidth (°) athwartship: Port/stbd offset = XXX degrees (to be subtracted from EK60 angles) (line #31).

o   RMS deviation (dB): RMS of fit to beam model out to XXX degrees = XXX dB (line #35).

Finally, set the values for the Gain (dB) and Sa Corr (dB) parameters in the fields provided in the Calibration tab.
